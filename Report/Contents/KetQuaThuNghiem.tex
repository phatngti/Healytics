\section{Kết quả thử nghiệm \& đánh giá}

\subsection{Kiểm thử chức năng (Functional Testing)}

Phần này trình bày kết quả kiểm thử chức năng của hệ thống, bao gồm kiểm thử đơn vị và kiểm thử tích hợp.

\subsubsection{Kiểm thử đơn vị (Unit Tests)}

Kiểm thử đơn vị nhằm đảm bảo từng thành phần nhỏ (Controller, Service) hoạt động đúng khi được tách riêng khỏi phần còn lại của hệ thống. Hệ thống sử dụng \textbf{Jest} làm công cụ kiểm thử chính, với tổng cộng \textbf{23 tệp kiểm thử} bao phủ toàn bộ các module:

\begin{itemize}
    \item \texttt{auth}: Kiểm tra đăng ký, đăng nhập, làm mới token và xác thực JWT.
    \item \texttt{partners}: Kiểm tra đăng ký đối tác, cập nhật hồ sơ, chuyển đổi trạng thái xác minh.
    \item \texttt{admin}: Kiểm tra quy trình xét duyệt, xác định kết quả duyệt và ghi nhận lịch sử xét duyệt.
    \item \texttt{employees}, \texttt{products}, \texttt{categories}, \texttt{locations}, \texttt{service-tags}, \texttt{s3}: Kiểm tra các thao tác thêm/sửa/xóa và logic nghiệp vụ.
\end{itemize}

Để cô lập từng thành phần, các bài kiểm thử sử dụng đối tượng giả lập (mock) thay thế cho cơ sở dữ liệu thực. Mỗi bài kiểm thử tuân theo mô hình 3 bước: \textbf{Chuẩn bị} dữ liệu đầu vào, \textbf{Thực thi} hàm cần kiểm tra, và \textbf{Xác nhận} kết quả đầu ra.

\subsubsection{Kiểm thử tích hợp (End-to-End Tests)}

Kiểm thử tích hợp mô phỏng hành vi thực tế của người dùng bằng cách gửi các yêu cầu HTTP đến hệ thống đang chạy đầy đủ. Các bài kiểm thử được tổ chức trong thư mục \texttt{test/e2e/}, sử dụng thư viện \texttt{Supertest}:

\begin{itemize}
    \item \texttt{auth/}: Kiểm tra luồng hoàn chỉnh từ đăng ký $\rightarrow$ đăng nhập $\rightarrow$ làm mới token.
    \item \texttt{account/}: Kiểm tra cập nhật hồ sơ cá nhân.
    \item \texttt{categories/}, \texttt{employees/}, \texttt{products/}: Kiểm tra các API thêm/sửa/xóa từ đầu đến cuối.
\end{itemize}

Để hỗ trợ việc viết kiểm thử nhanh và nhất quán, hệ thống cung cấp các thư mục tiện ích:

\begin{itemize}
    \item \texttt{test/factories/}: Tạo dữ liệu mẫu tự động cho các bài kiểm thử.
    \item \texttt{test/fixtures/}: Dữ liệu cố định dùng chung.
    \item \texttt{test/mocks/}: Các đối tượng giả lập dùng chung giữa nhiều bài kiểm thử.
    \item \texttt{test/helpers/}: Các hàm tiện ích hỗ trợ khởi tạo và dọn dẹp môi trường kiểm thử.
\end{itemize}

\subsubsection{Công cụ và lệnh kiểm thử}

\begin{itemize}
    \item \textbf{Chạy kiểm thử đơn vị:} \texttt{npm test}
    \item \textbf{Chạy kiểm thử tích hợp:} \texttt{npm run test:e2e}
    \item \textbf{Xem độ bao phủ mã nguồn:} \texttt{npm run test:cov}
    \item \textbf{Kiểm tra chất lượng mã:} \texttt{npx eslint src/}
    \item \textbf{Định dạng mã nguồn:} \texttt{npx prettier --write src/}
\end{itemize}

\paragraph{Flow kiểm thử xác thực và phân quyền}
Mục tiêu của kiểm thử này là trả lời ba câu hỏi: (1) API đăng nhập có hoạt động thật trên server triển khai hay không; (2) token trả về có phải JWT hay không; (3) backend có thật sự dùng vai trò trong token để chặn truy cập trái quyền hay không. Toàn bộ kiểm thử được thực hiện trực tiếp trên server qua API Gateway. Lưu ý rằng backend production/dev được Kong expose dưới prefix \texttt{/backend}; vì vậy URL đúng có dạng \texttt{https://healytics.me/backend/...} hoặc \texttt{https://dev.healytics.me/backend/...}. Nếu gọi thiếu prefix, ví dụ \texttt{https://dev.healytics.me/auth/user/login}, Kong sẽ trả \texttt{HTTP 404} với thông báo \texttt{no Route matched with those values}; đây là lỗi định tuyến gateway, không phải lỗi xác thực của backend.

\begin{enumerate}
    \item \textbf{Kiểm tra server đang hoạt động:} gọi \texttt{GET https://healytics.me/backend/health}. Kết quả trả về \texttt{HTTP 200 OK} cùng body \texttt{\{"status":"ok","service":"healytics-backend",...\}}, chứng tỏ backend thật đang chạy sau gateway.
    \item \textbf{Gọi API đăng nhập bằng curl:} gọi lần lượt ba endpoint \texttt{POST /backend/auth/user/login}, \texttt{POST /backend/auth/admin/login}, \texttt{POST /backend/auth/partner/login} với tài khoản seed tương ứng. Cả ba request đều trả về \texttt{HTTP 200} và body gồm \texttt{access\_token}, \texttt{refresh\_token}, \texttt{access\_expires\_in}, \texttt{refresh\_expires\_in}.
    \item \textbf{Kiểm tra token có phải JWT hay không:} một JWT hợp lệ có dạng ba đoạn phân tách bởi dấu chấm: \texttt{header.payload.signature}. Token trả về từ API đều có đúng dạng này, ví dụ bắt đầu bằng \texttt{eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...}.
    \item \textbf{Decode JWT để đọc vai trò:} lấy \texttt{access\_token} đưa vào jwt.io hoặc tự tách token theo dấu chấm rồi Base64URL-decode phần \texttt{header} và \texttt{payload}. Phần \texttt{header} cho biết thuật toán ký là \texttt{HS256}. Phần \texttt{payload} chứa các claim như \texttt{sub} (ID tài khoản), \texttt{email}, \texttt{role}, \texttt{iat} (thời điểm cấp), \texttt{exp} (thời điểm hết hạn).
    \item \textbf{Kiểm tra RBAC bằng truy cập chéo:} dùng token của từng vai trò gọi API không thuộc quyền của vai trò đó. Nếu RBAC hoạt động đúng, backend phải trả \texttt{403 Forbidden} cho truy cập trái quyền và \texttt{200 OK} cho truy cập đúng quyền.
\end{enumerate}

Kết quả decode cho thấy token user có \texttt{role = user}; token admin có \texttt{role = admin}; token partner có \texttt{role = health\_partner}, đồng thời có thêm \texttt{verificationStatus = APPROVED} và \texttt{partnerProfileCompleted = true}. Điều này chứng minh token không chỉ là chuỗi ngẫu nhiên mà chứa thông tin định danh và phân quyền được backend ký lại.

Kết luận: cơ chế xác thực và phân quyền hoạt động đúng ở các luồng đại diện. API login trả về JWT thật; JWT chứa claim vai trò rõ ràng; backend chặn truy cập trái quyền bằng \texttt{401/403} và cho phép truy cập đúng quyền bằng \texttt{200}.

\paragraph{Cách decode JWT}
JWT gồm ba phần \texttt{header.payload.signature}. \texttt{Header} mô tả thuật toán ký token, ví dụ \texttt{HS256}. \texttt{Payload} là dữ liệu đã mã hóa Base64URL, không phải dữ liệu bí mật, nên có thể decode để đọc các claim như \texttt{email} và \texttt{role}. \texttt{Signature} dùng secret phía server để đảm bảo token không bị sửa. Nếu attacker tự sửa \texttt{role} từ \texttt{user} thành \texttt{admin}, chữ ký sẽ không còn hợp lệ và backend phải từ chối token.

Quy trình decode thủ công tương đương jwt.io:
\begin{enumerate}
    \item Copy \texttt{access\_token} từ response login.
    \item Tách token thành ba phần theo dấu chấm.
    \item Base64URL-decode phần thứ nhất để đọc \texttt{header}.
    \item Base64URL-decode phần thứ hai để đọc \texttt{payload}.
    \item Đọc trường \texttt{role} trong payload để xác định tài khoản là \texttt{user}, \texttt{admin} hay \texttt{health\_partner}.
\end{enumerate}

\paragraph{Đánh giá bảo vệ API trước SQL Injection, XSS và CSRF}
\textbf{SQL Injection} là kỹ thuật chèn câu lệnh SQL độc hại vào input như email, tham số tìm kiếm hoặc ID để làm sai logic truy vấn database. Ví dụ trong màn hình đăng nhập, attacker có thể thử nhập email \texttt{' OR '1'='1} với mục tiêu làm điều kiện SQL luôn đúng và đăng nhập không cần mật khẩu thật. Trong kiểm thử này, payload trên được gửi trực tiếp đến \texttt{POST /backend/auth/user/login}; kết quả backend trả \texttt{HTTP 401 Unauthorized} với thông báo sai email hoặc mật khẩu. Như vậy, endpoint login không bị bypass bởi payload SQL Injection cơ bản.

Ngoài kiểm thử bằng payload, việc đọc hiện thực backend cho thấy hệ thống dùng TypeORM repository/query builder với tham số ràng buộc như \texttt{:partnerId}, \texttt{:status}, \texttt{:currency}, thay vì nối trực tiếp chuỗi input vào SQL. Backend cũng bật \texttt{ValidationPipe} với \texttt{whitelist}, giúp loại bỏ field ngoài DTO. Kết luận thận trọng: hệ thống đang phòng SQL Injection tốt ở endpoint login và các truy vấn đại diện đã xem; tuy nhiên chưa thể khẳng định tuyệt đối cho toàn bộ hệ thống nếu chưa pentest toàn bộ các endpoint search, filter, sort và analytics có raw SQL.

\textbf{XSS} là tấn công chèn HTML/JavaScript độc hại vào dữ liệu được hệ thống lưu và hiển thị lại cho người dùng khác. Các điểm có nguy cơ trong Healytics là tên/mô tả dịch vụ do partner nhập, nội dung review, tin nhắn chat, thông báo, hoặc thông tin hồ sơ phòng khám. Payload đại diện thường dùng là \texttt{<script>alert(1)</script>} hoặc \texttt{<img src=x onerror=alert(1)>}. Trong phạm vi kiểm thử hiện tại, chưa thực hiện stored XSS qua UI thật vì cần thao tác tạo dữ liệu và quan sát màn hình render lại. Tuy nhiên response từ server có header \texttt{content-security-policy} và các header Helmet khác, đây là lớp giảm thiểu XSS ở phía trình duyệt. Kết luận thận trọng: hệ thống có cơ chế giảm rủi ro XSS qua CSP/Helmet.

\textbf{CSRF} là tấn công lợi dụng trình duyệt của người dùng đã đăng nhập để gửi request thay đổi trạng thái mà người dùng không chủ ý. CSRF thường nguy hiểm với hệ thống dùng cookie session, vì cookie tự động được trình duyệt gửi kèm request. Healytics sử dụng JWT qua header \texttt{Authorization: Bearer <token>}; website độc hại bên ngoài không thể tự gắn header này nếu không biết token. Trong kiểm thử phân quyền, các API protected đều yêu cầu Bearer token hợp lệ; token sai vai trò bị trả \texttt{403}. Vì vậy rủi ro CSRF trong kiến trúc hiện tại thấp hơn cookie session truyền thống. Kết luận thận trọng: cơ chế Bearer JWT giúp phòng CSRF hiệu quả ở mức thiết kế; nếu sau này chuyển sang lưu token trong cookie, cần bổ sung CSRF token và cấu hình cookie \texttt{SameSite}, \texttt{Secure}, \texttt{HttpOnly}.

\subsubsection{Cơ sở lựa chọn ngưỡng đánh giá}

Các ngưỡng định lượng trong chương này không được chọn ngẫu nhiên, mà được thiết lập từ bốn nguồn chính nhằm bảo đảm tính khoa học và tính khả thi triển khai:

\begin{itemize}
    \item \textbf{Từ đặc tả yêu cầu hệ thống (SRS):} các ngưỡng đóng vai trò tiêu chí nghiệm thu (acceptance criteria) cho từng thành phần. Ví dụ, yêu cầu thời gian phản hồi API và yêu cầu chất lượng AI được gắn trực tiếp với mục tiêu vận hành của hệ thống.
    \item \textbf{Từ nguyên lý UX/HCI:} với các chỉ số độ trễ, nhóm tham chiếu nguyên lý thời gian phản hồi của Nielsen Norman Group (mức dưới 100--200ms tạo cảm giác gần thời gian thực), do đó đặt mục tiêu \texttt{Latency P95} ở mức chặt chẽ để bảo toàn trải nghiệm hội thoại.
    \item \textbf{Từ baseline học thuật và thực nghiệm:} các chỉ số như F1, Hit@K, MRR, NDCG được đối chiếu với mô hình cơ sở trên cùng bài toán; ngưỡng được đặt theo hướng mô hình đề xuất phải vượt baseline để chứng minh cải thiện thực chất.
    \item \textbf{Từ heuristic kỹ thuật theo bối cảnh dữ liệu:} với dữ liệu tiếng Việt tự do và truy vấn hội thoại có nhiễu, nhóm dùng quy tắc thực hành phổ biến trong NLP/IR để đặt ngưỡng khả thi nhưng đủ khắt khe (mức tốt thường từ 0.85 đến 0.90 trở lên tùy chỉ số).
\end{itemize}

Vì vậy, khi giải trình trước hội đồng, có thể nêu rằng các ngưỡng trong báo cáo được tổng hợp từ \textbf{yêu cầu hệ thống}, \textbf{chuẩn trải nghiệm người dùng}, \textbf{kết quả baseline}, và \textbf{kinh nghiệm thực nghiệm} trên dữ liệu của đề tài, thay vì dựa vào một con số cứng duy nhất cho mọi bài toán.

\noindent\textbf{Lựa chọn ngưỡng cho RAG và LLM trong module evaluation:}
\begin{itemize}
    \item \textbf{Đối với RAG (retrieval):} mục tiêu kỹ thuật của pha truy xuất là bảo đảm \textit{đưa ít nhất một bằng chứng đúng vào Top-$K$} và đồng thời \textit{phủ đủ các tài liệu liên quan} để phục vụ trả lời. Vì vậy nhóm dùng các chỉ số chuẩn trong Information Retrieval:
    \begin{itemize}
        \item \textbf{Hit@K}: đo khả năng ``có ít nhất một tài liệu đúng'' trong Top-$K$; phù hợp với tình huống hội thoại vì chỉ cần một bằng chứng tốt đã có thể hỗ trợ trả lời.
        \item \textbf{Recall@K}: đo tỷ lệ tài liệu liên quan được truy xuất; phản ánh mức độ bao phủ ngữ cảnh khi một câu hỏi có nhiều bằng chứng.
        \item \textbf{MRR}: nhấn mạnh vị trí xuất hiện đầu tiên của tài liệu đúng; trực tiếp liên quan đến chất lượng trải nghiệm vì bằng chứng đúng càng nằm sớm thì prompt càng ``sạch'' và giảm nhiễu cho LLM.
    \end{itemize}
    Trong pipeline đánh giá, $K$ được cố định theo tham số cấu hình \texttt{TOP\_K} (mặc định 10). Việc chọn $K$ theo hướng cân bằng giữa \textit{độ phủ} (tăng $K$) và \textit{giảm nhiễu/ngữ cảnh thừa} (giảm $K$), đồng thời nhất quán với cấu hình vận hành để số liệu phản ánh đúng điều kiện triển khai.

    \item \textbf{Đối với LLM (generation + grounding):} chất lượng câu trả lời được đánh giá theo thang điểm liên tục trong đoạn $[0,1]$ trên ba trục:
    \begin{itemize}
        \item \textbf{Faithfulness}: mức độ câu trả lời được \textit{ngữ cảnh truy xuất} hỗ trợ, nhằm kiểm soát rủi ro ``ảo giác'' (hallucination).
        \item \textbf{Correctness}: mức độ phù hợp về \textit{nghĩa} so với đáp án chuẩn (\textit{ground truth}), không yêu cầu khớp từng chữ.
        \item \textbf{Context Relevance}: mức độ ngữ cảnh truy xuất \textit{thực sự hữu ích} cho câu hỏi; chỉ số này giúp tách bạch lỗi do truy xuất (context kém liên quan) khỏi lỗi do sinh (answer kém).
    \end{itemize}
    Vì các điểm số này được chuẩn hóa về $[0,1]$, ngưỡng nghiệm thu (nếu đặt) có thể được diễn giải minh bạch theo ``mức đạt tối thiểu'' và ``mức kỳ vọng''; đồng thời việc dùng thang liên tục cho phép so sánh các biến thể mô hình/cấu hình mà không phụ thuộc vào một ranh giới cứng duy nhất. Trong luận văn này, cách đặt ngưỡng cho nhóm chỉ số LLM được ưu tiên theo hướng \textit{đảm bảo tính đúng và bám ngữ cảnh} (faithfulness/correctness) hơn là tối ưu văn phong.
\end{itemize}

\subsubsection{Hệ thống RAG (Retrieval-Augmented Generation)}

\textbf{Mục tiêu đánh giá.} Hệ thống RAG được đánh giá theo đúng hai thành phần chức năng cốt lõi của RAG: 
\begin{itemize}
    \item \textit{khả năng truy xuất} đưa đúng bằng chứng vào Top-$K$ (retrieval quality), 
    \item \textit{chất lượng ngữ cảnh} sau truy xuất khi được dùng làm đầu vào cho mô hình sinh. 
\end{itemize}
Trong phần này tập trung vào "khả năng truy xuất" thông qua các chỉ số chuẩn của IR; phần "chất lượng ngữ cảnh" được phản ánh gián tiếp thông qua \textit{Context Relevance} ở mục đánh giá LLM.

\textbf{Chuẩn bị dữ liệu đánh giá.} Bộ dữ liệu được đóng gói trong một tệp JSON theo cấu trúc thống nhất:
\begin{itemize}
    \item \texttt{documents}: danh sách tài liệu nguồn dùng để xây dựng VectorDB, mỗi phần tử có dạng \{\texttt{doc\_id}, \texttt{content}\}.
    \item \texttt{eval\_dataset}: danh sách mẫu đánh giá, mỗi mẫu gồm:
    \begin{itemize}
        \item \texttt{id}: định danh câu hỏi.
        \item \texttt{query}: câu hỏi/nhu cầu thông tin của người dùng.
        \item \texttt{ground\_truth}: đáp án chuẩn (dùng cho đánh giá chất lượng sinh ở mục LLM).
        \item \texttt{relevant\_doc\_ids}: tập \texttt{doc\_id} liên quan (gold evidence) dùng làm nhãn cho bài toán truy xuất.
        \item \texttt{difficulty}: nhãn độ khó (easy/medium/hard) phục vụ phân tích theo nhóm (không ảnh hưởng trực tiếp đến công thức metric).
    \end{itemize}
\end{itemize}
Để bảo toàn tính tái lập (reproducibility), toàn bộ pipeline đọc dữ liệu nạp trực tiếp từ tệp JSON; đồng thời tạo ánh xạ \texttt{doc\_id $\rightarrow$ content} để tái dựng ngữ cảnh theo đúng văn bản gốc khi cần.

\textbf{Xây dựng bộ truy xuất (Retriever).} Từ danh sách \texttt{documents}, mỗi tài liệu được chuyển sang định dạng \texttt{LangChain Document} với:
\begin{itemize}
    \item \texttt{page\_content} = nội dung văn bản \texttt{content}.
    \item \texttt{metadata["doc\_id"]} = định danh \texttt{doc\_id} để truy vết và tính metric.
\end{itemize}
Embeddings sử dụng \texttt{HuggingFaceEmbeddings} với tham số thiết bị \texttt{EMBEDDING\_DEVICE} (CPU/GPU). Vector store có thể chọn \texttt{Chroma} hoặc \texttt{FAISS} theo \texttt{VECTORSTORE\_BACKEND}. Khi khả dụng, pipeline ưu tiên dùng wrapper \texttt{VectorDB} trong \texttt{src.rag.vectorstore} để đồng nhất với hiện thực hệ thống; nếu không, sẽ fallback sang backend vector store trực tiếp. Retriever được cấu hình \texttt{search\_kwargs=\{"k": TOP\_K\}} để bảo đảm số lượng tài liệu truy xuất đúng với tham số đánh giá.

\textbf{Luồng đánh giá truy xuất.} Với mỗi mẫu \texttt{eval\_dataset}, pipeline thực hiện:
\begin{itemize}
    \item Nhận \texttt{query} và tập nhãn \texttt{relevant\_doc\_ids}.
    \item Gọi retriever để lấy danh sách tài liệu liên quan. 
    \item Cắt Top-$K$ kết quả đầu và trích \texttt{doc\_id} từ \texttt{metadata} để tạo chuỗi \texttt{retrieved\_doc\_ids}.
    \item Tính các metric IR và cộng dồn theo mẫu; cuối cùng lấy trung bình theo số mẫu.
\end{itemize}

\textbf{Chỉ số đánh giá và định nghĩa.} Gọi \(R_K\) là danh sách \texttt{doc\_id} truy xuất trong Top-$K$ và \(G\) là tập \texttt{relevant\_doc\_ids}:
\begin{itemize}
    \item \textbf{Hit@K}: \(=1\) nếu tồn tại \(d \in R_K\) sao cho \(d \in G\); ngược lại \(=0\).
    \item \textbf{Recall@K}: \(=\frac{|R_K \cap G|}{|G|}\) (nếu \(G=\varnothing\) thì quy ước \(=0\)).
    \item \textbf{MRR}: nếu tài liệu đúng đầu tiên xuất hiện ở vị trí \(i\) trong danh sách truy xuất thì \(MRR=\frac{1}{i}\); nếu không có tài liệu đúng thì \(=0\).
\end{itemize}
Các chỉ số trên phản ánh trực tiếp chất lượng truy xuất trong bối cảnh RAG: Hit@K đảm bảo ``có bằng chứng'', Recall@K đảm bảo ``đủ bằng chứng'', và MRR đảm bảo ``bằng chứng ưu tiên nằm sớm'' để tối ưu prompt và giảm nhiễu cho mô hình sinh.

\subsubsection{Mô hình LLM (Llama-3.1-8B-Instruct)}

\textbf{Mục tiêu đánh giá.} Mô hình LLM được đánh giá theo vai trò \textit{generator} trong pipeline RAG (sinh câu trả lời dựa trên truy vấn và ngữ cảnh truy xuất), đồng thời kiểm soát hai rủi ro chính: 
\begin{itemize}
    \item trả lời đúng nội dung so với đáp án chuẩn, 
    \item bám sát ngữ cảnh để hạn chế rủi ro hallucination. 
\end{itemize}
Vì đánh giá tự động theo luật/regex không phản ánh tốt các biến thể diễn đạt, pipeline sử dụng một \textit{LLM-as-a-judge} để chấm điểm mềm (continuous scoring) trên thang \( [0,1] \). Ở bước sinh, mỗi mẫu chỉ truyền payload tối thiểu gồm \texttt{question=query}, và để rỗng \texttt{history} cùng \texttt{services} nhằm cô lập tác động của truy xuất và mô hình sinh trong thí nghiệm.

\textbf{Chuẩn bị ngữ cảnh đưa vào Judge.} Sau khi retriever trả về Top-$K$ tài liệu, pipeline tái dựng văn bản ngữ cảnh theo thứ tự truy xuất và ghép bằng \texttt{"\textbackslash n\textbackslash n"}. Để tránh sai lệch do cơ chế lưu/truy hồi của vector store, pipeline ưu tiên lấy \texttt{content} theo ánh xạ \texttt{doc\_id $\rightarrow$ content} từ dataset gốc; chỉ khi không ánh xạ được mới dùng \texttt{page\_content} của đối tượng \texttt{Document}.

\textbf{Đánh giá tự động bằng Judge LLM (LLM-as-a-judge).} Một mô hình Judge độc lập được dùng để chấm ba tiêu chí:
\begin{itemize}
    \item \textbf{Faithfulness}: mức độ câu trả lời được ngữ cảnh hỗ trợ.
    \item \textbf{Correctness}: mức độ đúng về nghĩa so với \texttt{ground\_truth}.
    \item \textbf{Context Relevance}: mức độ ngữ cảnh liên quan và hữu ích cho câu hỏi.
\end{itemize}
Judge hoạt động theo hai chế độ:
\begin{itemize}
    \item \texttt{JUDGE\_BACKEND=mistral} (mặc định): gọi endpoint \texttt{https://api.mistral.ai/v1} theo chuẩn OpenAI-compatible với \texttt{JUDGE\_MODEL} (mặc định \texttt{mistral-large-latest}).
    \item \texttt{JUDGE\_BACKEND=same}: tái sử dụng chính generator LLM để làm judge (phục vụ thử nghiệm ablation).
\end{itemize}
Prompt chấm điểm yêu cầu Judge trả về \textbf{JSON} thuần với ba khóa \texttt{faithfulness}, \texttt{correctness}, \texttt{context\_relevance}. Để tăng độ bền khi LLM trả về thừa văn bản, pipeline trích xuất đối tượng JSON bằng regex, sau đó ép kiểu về số thực và \textit{clip} về đoạn \([0,1]\). Cuối cùng, các điểm số được \textbf{lấy trung bình theo số mẫu} để báo cáo \texttt{avg\_faithfulness}, \texttt{avg\_correctness}, \texttt{avg\_context\_relevance} cùng metadata của lần chạy (mode, model, backend, v.v.).

\textbf{Prompt đánh giá (nguyên văn).} Dưới đây là prompt được dùng để gọi Judge (Mistral) chấm điểm cho từng mẫu; prompt này được format theo các biến \texttt{\{question\}}, \texttt{\{context\}}, \texttt{\{answer\}}, \texttt{\{ground\_truth\}}:

\begin{verbatim}
You are a fair, lenient evaluator. Prefer partial credit: nếu câu trả lời nhìn chung đúng hoặc hợp lý so với ground truth thì cho điểm cao; khác từ ngữ nhỏ, thiếu chi tiết phụ, hoặc diễn đạt khác nhưng không sai nghĩa thì không trừ mạnh.

Given:
- Question
- Context
- Answer
- Ground truth

Score each metric in [0, 1]:
- Faithfulness: answer được context hỗ trợ; nếu có thêm suy luận nhẹ nhưng không mâu thuẫn context thì vẫn có thể cao.
- Correctness: answer phù hợp nghĩa với ground truth (không cần khớp từng chữ).
- Context Relevance: context có giúp trả lời câu hỏi (dù không hoàn hảo vẫn có thể điểm khá).

Return JSON only with keys: faithfulness, correctness, context_relevance.

Question:
{question}

Context:
{context}

Answer:
{answer}

Ground truth:
{ground_truth}
\end{verbatim}

\textbf{Nguồn gốc các chỉ số thước đo:}
\begin{itemize}
    \item Hit@K: bài toán \textit{top-k retrieval/recommendation}.
    \item Precision@K, Recall@K: Information Retrieval (IR) và Recommendation evaluation.
    \item MRR@K: QA và ranking tasks.
    \item NDCG@K, NDCG\_G@K: Learning-to-Rank.
    \item HNR@K: thực hành \textit{hard-negative analysis} trong retrieval.
    \item Latency (Avg/P50/P95/P99/Max): SRE và phương pháp phân vị hiệu năng.
\end{itemize}

\textbf{Các chỉ số thước đo:}
\begin{itemize}
    \item \textbf{Hit@K}: đo khả năng có ít nhất một kết quả đúng trong Top-K; được chọn vì phản ánh trực tiếp trải nghiệm người dùng có tìm thấy gợi ý phù hợp hay không.
    \item \textbf{Precision@K}: đo độ chính xác trong danh sách trả về; được chọn để kiểm soát mức độ "loang" của kết quả khi tăng K.
    \item \textbf{Recall@K}: đo độ bao phủ kết quả liên quan; được chọn vì hệ thống Home cần bao phủ đa nhu cầu của hồ sơ người dùng.
    \item \textbf{MRR@K}: nhấn mạnh vị trí của kết quả đúng đầu tiên; được chọn vì chatbot cần đưa đúng gợi ý lên rất sớm để giảm thao tác người dùng.
    \item \textbf{NDCG@K, NDCG\_G@K}: đánh giá chất lượng thứ hạng có xét vị trí và mức độ liên quan; được chọn để đo chất lượng ranking tổng thể thay vì chỉ kiểm tra có/không có kết quả đúng.
    \item \textbf{HNR@K} ($\downarrow$): đo tỷ lệ nhiễu lọt Top-K; được chọn để theo dõi rủi ro gợi ý sai ngữ nghĩa ở các vị trí đầu.
    \item \textbf{Latency (Avg/P50/P95/P99/Max)}: phản ánh cả hiệu năng trung bình lẫn tail-latency, trong đó P95 là chỉ số chính cho trải nghiệm thực tế ổn định.
\end{itemize}

\textbf{Lựa chọn ngưỡng triển khai:}

\end{itemize}

\subsubsection{Đánh giá End-to-End toàn bộ luồng AI (LLM-as-a-Judge)}

\textbf{Mục tiêu đánh giá.}
Các phần trước đã đánh giá từng thành phần độc lập: truy xuất của \texttt{RAG}, chất lượng sinh của \texttt{LLM} và hệ thống gợi ý (\texttt{Recommender}).
Mục này đánh giá \textbf{toàn bộ luồng AI ở mức end-to-end}: từ khi người dùng gửi truy vấn tới khi nhận được câu trả lời cuối cùng, bao gồm tất cả các bước điều phối (intent, NER, retrieval, prompt enrichment, LLM generation, streaming).
Khác với đánh giá từng thành phần, mục tiêu ở đây là trả lời câu hỏi: \emph{câu trả lời mà hệ thống đưa tới tay người dùng cuối có thực sự đúng, hữu ích và bám sát câu hỏi hay không}.
Do không có nhãn gold chuẩn cho câu trả lời sinh tự do, chúng tôi áp dụng phương pháp \textbf{LLM-as-a-Judge} (Zheng et al., 2023) với \texttt{Mistral} làm mô hình chấm điểm độc lập (không trùng với \texttt{LLM} đang sinh trong hệ thống, nhằm giảm thiên lệch tự chấm).

\textbf{Mô hình sinh được đánh giá.}

\textbf{Judge LLM và prompt đánh giá.}
Chúng tôi dùng \texttt{mistral-large-latest} (API Mistral) với \texttt{temperature=0} để đảm bảo tính tái lập. Với mỗi mẫu, Judge nhận 3 thành phần (\textit{Question}, \textit{Ground truth}, \textit{Model answer}) và chấm điểm đồng thời 3 tiêu chí trên thang \textbf{1--5}, kèm giải thích bằng tiếng Việt. Prompt được dùng như sau:

\begin{verbatim}
You are an evaluator for a healthcare chatbot.

Given:
- User query
- Ground truth answer
- Model answer

Evaluate the model answer based on:

1. Correctness (Is the answer factually correct?)
2. Helpfulness (Is the answer useful to the user?)
3. Relevance (Does the answer address the query?)

Score each from 1 to 5.

Also provide a brief explanation.

Return JSON only, exactly in this schema:
{
  "correctness": <int 1-5>,
  "helpfulness": <int 1-5>,
  "relevance":   <int 1-5>,
  "explanation": "<short explanation in Vietnamese>"
}
\end{verbatim}

\textbf{Cơ sở lựa chọn ngưỡng.}
Mỗi tiêu chí được chấm theo thang rời rạc 1--5, trong đó $\ge 4$ được coi là \textit{đạt chất lượng cao} (câu trả lời đúng, hữu ích, bám sát câu hỏi), $=3$ là \textit{chấp nhận được} (không sai nhưng chưa tối ưu), $\le 2$ là \textit{chưa đạt}. Vì vậy ngoài điểm trung bình chúng tôi báo cáo thêm \textbf{phân phối điểm 1--5} và \textbf{pass rate} (tỉ lệ mẫu đạt $\ge 4$). Các mục tiêu ngưỡng ban đầu cho phiên bản cơ sở: \texttt{Correctness\_mean} $\ge 3.0$, \texttt{Helpfulness\_mean} $\ge 3.0$, \texttt{Relevance\_mean} $\ge 3.5$ và \texttt{Relevance pass rate} $\ge 50\%$.

\subsubsection{System Metrics (Latency, Throughput, Cost)}

\textbf{Mục tiêu.}
Ngoài chất lượng sinh, khả năng phục vụ thực tế của hệ thống còn phụ thuộc vào hiệu năng (độ trễ), khả năng thông lượng (throughput) và chi phí vận hành (cost). Mục này đo ba nhóm chỉ số này trên cùng bộ 500 truy vấn end-to-end đã dùng ở mục trước, để có một cái nhìn thống nhất giữa \emph{chất lượng} và \emph{hiệu năng--chi phí}.

\textbf{Các chỉ số được đo.}
\begin{itemize}
    \item \textbf{First Token Latency (TTFT)}: khoảng thời gian từ lúc client gửi request tới lúc nhận được token đầu tiên. Đây là chỉ số quan trọng nhất đối với trải nghiệm ``phản hồi cảm nhận được'' của người dùng.
    \item \textbf{Total Latency}: thời gian từ lúc gửi request tới lúc nhận xong toàn bộ câu trả lời (stream đóng).
    \item \textbf{Streaming Time}: thời gian giữa first-token và last-token, biểu diễn thời lượng mà client thực sự nhận được các chunk token.
    \item \textbf{Throughput}: số từ trả về trung bình mỗi giây trong pha streaming (\texttt{answer\_word\_count / streaming\_time}).
    \item \textbf{Cost per Request}: ước lượng chi phí mỗi yêu cầu dựa trên thời gian chiếm dụng GPU khi chạy \texttt{meta-llama/Llama-3.1-8B-Instruct} trên server A5000 giá \textbf{\$0.27/giờ}, tính theo công thức:
    \[
    \text{Cost}_{\text{USD}} = \frac{\text{TotalLatency}_{\text{seconds}}}{3600} \times 0.27
    \]
\end{itemize}

\textbf{Cấu hình đo đạc.}
Mọi request được đo tuần tự (concurrency = 1) để phản ánh độ trễ cold-path của một người dùng. Hệ thống đo đạc dùng client HTTP/SSE bất đồng bộ (\texttt{httpx.AsyncClient}) ghi lại mốc thời gian ở mỗi sự kiện SSE (\texttt{token}, \texttt{recommendation}, \texttt{done}).

\noindent\textbf{Nhận xét về hiệu năng và chi phí:}
\begin{itemize}
    \item \textbf{Độ trễ giảm đáng kể} so với phiên bản Llama-3B cục bộ: Total Latency trung bình từ 87.8s xuống còn 8.47s ($\approx$10 lần), P95 từ 188.75s xuống 17.92s. Đây là kết quả trực tiếp từ việc triển khai \texttt{meta-llama/Llama-3.1-8B-Instruct} trên GPU A5000 kết hợp với true-streaming.
    \item \textbf{TTFT trung bình 1.21 giây} đạt ngưỡng trải nghiệm tốt cho chatbot ($<$2s), người dùng bắt đầu thấy phản hồi gần như ngay lập tức.
    \item \textbf{P95 Total Latency = 17.92s} là mức chấp nhận được cho chatbot y tế, nơi câu trả lời dài và chính xác quan trọng hơn tốc độ tuyệt đối.
    \item \textbf{Throughput tăng từ 59.6 lên 87.3 từ/giây} (+46\%), phản ánh tốc độ sinh token cao hơn của GPU A5000 so với GPU cục bộ RTX 3050 Laptop.
    \item \textbf{Chi phí trung bình thấp}: với giá GPU A5000 \textbf{\$0.27/giờ}, chi phí suy luận trung bình là \textbf{\$0.00064/request}; với 100.000 request/ngày, chi phí ước tính khoảng \textbf{\$63.5/ngày} nếu mở rộng tuyến tính theo thời gian xử lý trung bình.
    \item \textbf{Nhóm \texttt{greeting\_smalltalk}} có cost chỉ \textbf{\$0.00024} (khoảng 1/3--1/4 các nhóm còn lại), cho thấy hướng tối ưu hóa phân luồng theo intent vẫn còn giá trị để giảm thêm chi phí.
\end{itemize}

\textbf{Kết luận hiệu năng--chi phí.}
Hệ thống với \texttt{meta-llama/Llama-3.1-8B-Instruct} trên GPU A5000 đạt hiệu năng vượt trội cả về chất lượng lẫn tốc độ so với phiên bản Llama-3.2-3B: Total Latency giảm 10 lần, TTFT đạt $<$1.5s trung bình, throughput tăng 46\%, trong khi chi phí suy luận trung bình chỉ khoảng \textbf{\$0.00064/request}. Đây là cấu hình đủ điều kiện triển khai production với chi phí hạ tầng khoảng \textbf{\$6.48/ngày cho mỗi GPU A5000 chạy liên tục}, tương đương khoảng \textbf{\$63.5/ngày} cho quy mô 100.000 request nếu mở rộng tuyến tính theo kết quả đo tuần tự.
